Requirements & Implementation

Q1: What were the core requirements for the medium task?

Minimum requirements:

  • Interactive Leaflet map showing wildlife sightings
  • Organism selector with filters (state, month, record type)
  • Basic visualizations of sighting patterns
  • Weather data integration (optional bonus)

Implementation delivered:

  • 4-tab interface (Explorer, Predictions, Tourism, Time Analysis)
  • Leaflet map with marker clustering for performance
  • Complete weather analysis (boxplots, scatter plots, temperature distributions)
  • GLM-based sighting predictions (Poisson/Negative Binomial models)
  • Tourism region analysis with state breakdowns
  • Comprehensive time analysis (hourly, weekday, hour×month heatmap)
  • 5 KPI summary cards
  • Download handlers for CSV and PNG exports
  • Robust error handling and validation
  • Professional design with large, readable fonts
  • Full mobile responsiveness
Q2: What technical architecture did you choose and why?

Initial Implementation (v1):

  • Custom JavaScript for tab switching
  • Complex CSS with nested selectors (~700 lines)
  • Manual show/hide logic for tab panels
  • Small fonts (0.78rem - 1rem)

Issues with v1:

  • Tabs not rendering (Predictions, Tourism, Time Analysis blank)
  • Fonts too small for readability
  • Custom JS caused Shiny output suspension bugs
  • Hard to maintain and debug

Final Implementation (v2 - Complete Rewrite):

  • Native Shiny tabsetPanel (no custom JS)
  • Simplified CSS (~500 lines, clean structure)
  • Large fonts throughout (18px base, 1.1rem+ labels, 2.2rem KPIs)
  • All tabs render instantly
  • ~850 lines of R code (down from 1136 lines)
  • Better error handling and validation

Features Implementation

Q4: How does each tab work?

Tab 1: Explorer

Purpose: Main data exploration interface

Components:

  • Interactive Map:
    • Leaflet with CartoDB.Positron tiles
    • Marker clustering (essential for Orchids with 300k+ records)
    • Circle markers colored by species
    • Popups show: species, date, state, hour, record type
  • Weather Analysis:
    • Boxplot: Temperature distribution by month
    • Shows seasonal patterns at sighting times
  • Monthly Bar Chart:
    • Counts by month with season colors
    • Peak month highlighted in subtitle
    • Gradient fill (light to dark green)
  • Annual Trend:
    • Area chart + line + points
    • Shows multi-year patterns

Tab 2: Predictions

Purpose: Forecast best upcoming days for wildlife viewing

How it works:

  1. User sets N days to predict (1-20)
  2. Click "Generate Predictions"
  3. Algorithm:
    • Find busiest weather station
    • Join occurrence counts with weather data
    • Fit GLM (Poisson or Negative Binomial if overdispersed)
    • Project forward using climatological normals
    • Rank by predicted count with 95% CI
  4. Display:
    • Model stats (type, observations, AIC)
    • Table: Top N days with dates, best hour, temp, confidence intervals
    • Chart: Visual comparison with error bars

Error handling:

  • Before first click: Info message prompting user
  • Insufficient data: Warning with suggestion to widen filters
  • Success: Full results with model diagnostics

Tab 3: Tourism

Purpose: Analyze geographic distribution

Components:

  • State Chart:
    • Horizontal bar chart of sightings by state
    • Point size indicates percentage
    • Labels show count and percentage
  • Regional Table:
    • Breakdown by state/territory
    • Columns: Sightings, Species, First/Last year, % of total
    • Sortable and styled

Tab 4: Time Analysis

Purpose: Understand temporal patterns

Components:

  • Hourly Distribution:
    • Bar chart showing sightings by hour (0-23)
    • Colored by time period (Night/Morning/Afternoon/Evening)
    • Peak hour highlighted in subtitle
  • Weekday Distribution:
    • Column chart for each day of week
    • Weekends in orange, weekdays in green
    • Percentages labeled on bars
  • Hour×Month Heatmap:
    • Tile plot: month (rows) × hour (columns)
    • Viridis color gradient for count intensity
    • Reveals best times to spot wildlife by season
Q5: How did you handle edge cases and errors?

Validation implemented:

# Empty filter results
validate(need(nrow(d) > 0,
  "No sightings match filters. Try widening your selection."))

# Insufficient weather data
validate(need(nrow(w) > 5,
  "Not enough weather data for visualization."))

# Missing hour data
validate(need(nrow(d) > 10,
  "Not enough hour data for heatmap."))

Error messages are user-friendly:

  • Clear explanation of what went wrong
  • Actionable suggestion (e.g., "Try 'All Months'")
  • No technical jargon or R error traces
  • Color-coded info boxes (blue=info, orange=warning)

Prediction failures handled gracefully:

  • If data < 30 rows: "Not enough data for predictions"
  • If station has < 5 sightings: Automatic fallback to best station
  • If weather data missing: Clear error message
  • Model type automatically selected (Poisson vs NegBin)

Data Management

Q6: How is data loaded and managed?

Data files (local RDS for deployment):

medium-task/data/
├── gouldian_finch.rds    (3,921 records)
├── manta_rays.rds        (1,204 records)
├── glowworms.rds         (847 records)
├── orchids.rds           (302,714 records)
├── weather.rds           (458,137 records)
└── tourism_region.rds    (regional mappings)

Why local RDS files?

  • Faster deployment (no package dependency resolution)
  • Guarantees exact data versions
  • Works on shinyapps.io free tier
  • 600KB compressed - quick cold start

Reactive data pipeline:

organism_data (list)
    ↓
base_data() — filters by organism
    ↓
filtered_data() — applies state, month, record type filters
    ↓
with_weather() — left join with weather data
    ↓
Charts, map, tables

Performance optimization:

  • Reactive chain prevents unnecessary recalculation
  • Map uses leafletProxy for efficient marker updates
  • Marker clustering prevents DOM overload with large datasets
  • Plot builders reused between render and download handlers

Lessons Learned

Q10: What did you learn from this task?

Technical Lessons

  • Native over custom: Shiny's built-in components are well-tested and reliable. Custom JavaScript solutions add complexity and bugs.
  • Accessibility matters: Font sizes that seem "okay" on your screen may be illegible to others.
  • Validation is critical: Never assume data exists. Every reactive should have validation.
  • Simplicity scales: The v2 rewrite removed 286 lines of code and works better.

Design Lessons

  • Readability first: Larger fonts improve user experience more than fancy styling.
  • Error messages are UX: Clear, actionable error messages prevent frustration.
  • Progressive disclosure: Don't render everything at once. Tabs help organize complexity.

Process Lessons

  • Test systematically: The 16-combination test matrix caught all edge cases.
  • Document decisions: CHANGES_SUMMARY.md helps explain the evolution.
  • Iterate based on feedback: The complete rewrite addressed all reported issues.
