Algorithm Design

The Challenge

How do we predict the best future days for wildlife sighting when we don't have future weather data? How do we account for the overdispersion typical in wildlife count data?

1

Select Station

Auto-select the weather station with the most sightings, or use user-specified station_id

2

Aggregate Daily Counts

Count sightings per day at the selected station. Days with no sightings become zeros.

3

Join with Weather

LEFT JOIN weather onto sighting counts so that days with weather but zero sightings are included (the model needs those zeros).

4

Fit GLM

Start with Poisson. Check dispersion ratio. If >1.5 (overdispersed), refit as Negative Binomial using MASS::glm.nb().

5

Build Future Grid

Create future dates. Approximate weather using climatological normals (historical averages by calendar day).

6

Predict & Rank

Predict counts on future grid, compute 95% CI, rank by predicted count descending, return top N days.

Question 1: How to handle overdispersion in wildlife counts?

The Challenge

Wildlife count data typically has more variance than a Poisson distribution predicts (overdispersion). Using Poisson when overdispersed leads to underestimated standard errors and overconfident predictions.

Our Approach

  1. Fit initial Poisson GLM: glm(count ~ weather, family = poisson())
  2. Calculate dispersion ratio: sum(pearson_resid²) / df.residual
  3. If ratio > 1.5, refit using Negative Binomial: MASS::glm.nb()
  4. Attach model type to results for transparency

Solution Implementation

The function automatically selects the appropriate model and reports which was used.

# Fit Poisson first fmla <- sighting_count ~ temp + max + prcp + rh + wind_speed + month_f fit_pois <- glm(fmla, data = model_df, family = poisson()) # Check dispersion disp_ratio <- sum(residuals(fit_pois, type = "pearson")^2) / fit_pois$df.residual if (disp_ratio > 1.5) { fit <- MASS::glm.nb(fmla, data = model_df) model_type <- "Negative Binomial" message("Overdispersion detected (ratio = ", round(disp_ratio, 2), "), using negative binomial") } else { fit <- fit_pois model_type <- "Poisson" }

Question 2: How to predict without future weather data?

The Challenge

Our model uses weather covariates, but we don't have actual future weather. Weather APIs only provide 7-14 day forecasts, not a full year.

Our Approach: Climatological Normals

  • Calculate historical averages for each calendar day (e.g., "average temp on March 15")
  • Use these as expected weather for future predictions
  • This is a standard climatological baseline approach
  • Could be replaced with forecast API for short-term (noted as limitation)

Solution Implementation

We compute historical means by month-day combination and join to future dates.

# Compute climatological normals climate_normals <- wx_stn |> mutate(m = month(date), d = day(date)) |> group_by(m, d) |> summarise( temp = mean(temp, na.rm = TRUE), max = mean(max, na.rm = TRUE), prcp = mean(prcp, na.rm = TRUE), rh = mean(rh, na.rm = TRUE), wind_speed = mean(wind_speed, na.rm = TRUE), .groups = "drop" ) # Build future grid future_grid <- tibble(date = future_dates) |> mutate(m = month(date), d = day(date)) |> left_join(climate_normals, by = c("m", "d"))

Question 3: How to make the function robust?

The Challenge

Users might pass incomplete data, invalid parameters, or trigger edge cases. The function should fail gracefully with clear error messages.

Our Approach: Comprehensive Validation

  • Check required columns exist in both inputs
  • Validate n_days and forecast_horizon are positive integers
  • Check for zero-row data frames
  • Verify minimum weather records (30) for model fitting
  • Handle NA values gracefully
  • Fix MASS/dplyr select() conflict explicitly

Solution: 27 Unit Tests

We wrote a comprehensive test suite covering all edge cases and validation scenarios.
